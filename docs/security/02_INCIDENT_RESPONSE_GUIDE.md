# Incident Response Guide

## Purpose

This guide explains what to do when KRAVIA Company OS has a security, access, data, document, or availability incident.

## Incident Examples

An incident may include:

- Lost or exposed password
- Wrong user access level
- Private document shared incorrectly
- Unexpected document download
- Suspicious login
- Missing or changed company records
- System outage
- Failed backup
- Data entered incorrectly in a sensitive record

## First Response

1. Stay calm.
2. Stop the immediate risk if possible.
3. Do not delete evidence.
4. Notify a founder.
5. Record what happened.
6. Record time, user, module, and affected data.
7. Check audit logs.
8. Decide whether access should be disabled or reduced.

## Account Compromise

Steps:

1. Ask the affected user to stop using the account.
2. Change password or disable access.
3. Review recent audit logs.
4. Review document downloads.
5. Review edits made by the account.
6. Restore or correct records if needed.
7. Re-enable access only after approval.

## Document Exposure

Steps:

1. Identify the document.
2. Identify who accessed or received it.
3. Stop further sharing.
4. Review audit logs.
5. Notify founder and legal advisor if required.
6. Record the incident.
7. Decide whether document access rules must change.

## Data Error

Steps:

1. Identify the incorrect record.
2. Do not delete the record unless policy allows it.
3. Correct the record with a clear note.
4. Confirm audit log records the change.
5. Notify people who relied on the incorrect information.

## System Outage

Steps:

1. Check whether frontend, backend, database, or storage is affected.
2. Check health endpoints if available.
3. Notify users if work should pause.
4. Preserve logs.
5. Restore service.
6. Confirm login, company profile, documents, and audit logs work.
7. Record the outage and resolution.

## Backup Failure

Steps:

1. Confirm the backup failure.
2. Do not ignore it.
3. Notify founder.
4. Run a new backup if safe.
5. Check storage and database access.
6. Record the failure and fix.
7. Schedule restore test if needed.

## Incident Record

Every incident record should include:

- Date and time
- Reported by
- Module affected
- What happened
- Users affected
- Data affected
- Immediate action taken
- Final resolution
- Follow-up actions

## After The Incident

1. Review root cause.
2. Update user roles if needed.
3. Update security rules if needed.
4. Add tasks for follow-up work.
5. Confirm the issue does not repeat.
